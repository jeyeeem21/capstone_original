1. REAL-TIME DATA SYNC (Visibility-Based Refetch)
================================================================================

PROBLEM:
  When Super Admin makes changes (e.g. adds a procurement), and Admin is signed
  in on another computer/browser, the Admin had to manually refresh to see the
  updated data. This is not acceptable for production.

SOLUTION:
  Added automatic refetch-on-tab-focus to the useDataFetch hook.

HOW IT WORKS:
  - Every active useDataFetch hook registers itself with a global listener.
  - When the user switches BACK to the browser tab (visibilitychange event)
    or the browser window gains focus (focus event), ALL active hooks check
    whether their cached data is older than 5 seconds.
  - If data is stale (>5s old), a silent background network request fetches
    fresh data automatically — no loading spinner, no user action needed.
  - The 5-second threshold prevents unnecessary refetches if the user rapidly
    switches between tabs.
  - Cleanup: when a component unmounts, its refetch callback is removed from
    the active hooks set — no memory leaks.

FILE MODIFIED:
  react_frontend/src/hooks/useDataFetch.js

  Added:
  - REFETCH_ON_FOCUS_STALE constant (5 seconds)
  - activeHooks Set (tracks all mounted hooks' refetch callbacks)
  - attachVisibilityListener() — one-time setup for visibilitychange + focus
  - New useEffect inside useDataFetch that registers/deregisters the hook

BEHAVIOR IN PRODUCTION:
  - Super Admin adds a procurement on Computer A.
  - Admin switches to their browser tab on Computer B.
  - Data automatically refreshes within milliseconds of tab focus.
  - Works for ALL pages that use useDataFetch (Dashboard, Procurement, Drying,
    Processing, Products, Varieties, Inventory, Sales, Orders, Partners,
    Supplier, Customer, StaffManagement, AuditTrail, Archives, Settings,
    PointOfSale, Client Shop, Client Orders, Driver Deliveries, etc.)
  - No WebSocket server or Pusher subscription needed — works with standard
    HTTP and the existing Laravel API.

2. ERROR BOUNDARY (Modular Error Isolation)
================================================================================

PROBLEM:
  If ONE page has a JavaScript error (e.g. a null reference, API format change,
  or component crash), the entire React app would white-screen. All pages would
  become inaccessible until refresh.

SOLUTION:
  Created a React Error Boundary component and wrapped EVERY route's page
  component individually with it.

HOW IT WORKS:
  - ErrorBoundary is a class component (React requirement for error catching)
    that implements componentDidCatch and getDerivedStateFromError.
  - Each page route in App.jsx is wrapped: <ErrorBoundary><Dashboard /></ErrorBoundary>
  - If a page throws an error, ONLY that page shows a fallback UI.
  - All other pages, sidebar navigation, and the layout remain fully functional.
  - The fallback UI shows:
    * "Something went wrong" message
    * "Try Again" button (resets error state, re-renders the component)
    * "Go Back" button (navigates to previous page)
    * "Home" button (navigates to home/login)
    * In development: error details and component stack trace for debugging

FILE CREATED:
  react_frontend/src/components/ErrorBoundary.jsx

FILE MODIFIED:
  react_frontend/src/App.jsx
    - Every <Route> element now wraps its page component in <ErrorBoundary>
    - Both /superadmin/* and /admin/* routes wrapped
    - /staff/*, /client/*, /driver/* routes wrapped
    - Public routes (Home, About, Products, Contact) wrapped

BEHAVIOR IN PRODUCTION:
  - If the Procurement page has an error, user sees a friendly error screen
    with recovery options. They can navigate to Dashboard, Products, or any
    other page — everything else keeps working.
  - If they click "Try Again", the Procurement page re-renders fresh.
  - No data loss on other pages.

3. 404 NOT FOUND PAGE
================================================================================

PROBLEM:
  Navigating to a non-existent URL (e.g. /admin/nonexistent) silently
  redirected to "/" with no feedback. Users had no idea what happened.

SOLUTION:
  Created a proper 404 page and added catch-all routes.

HOW IT WORKS:
  - A NotFound component displays "404 - Page Not Found" with the attempted URL.
  - Smart navigation: if authenticated, "Go to Dashboard" links to the user's
    role-specific dashboard. If not authenticated, links to Home.
  - "Go Back" button takes user to the previous page.
  - Catch-all routes (<Route path="*">) added to:
    * /superadmin/* (inside MainLayout — shows 404 within the admin interface)
    * /admin/* (inside MainLayout)
    * /staff/* (inside StaffLayout)
    * /client/* (inside ClientLayout)
    * /driver/* (inside DriverLayout)
    * Global catch-all (top-level — for completely unknown paths)

FILE CREATED:
  react_frontend/src/pages/NotFound.jsx

FILE MODIFIED:
  react_frontend/src/App.jsx
    - Added <Route path="*" element={<NotFound />} /> in each route group
    - Changed global catch-all from <Navigate to="/" /> to <NotFound />

14. PUBLIC PRODUCT CARD FIX (Field Mapping)
================================================================================

PROBLEM:
  Public Home and Products pages showed blank product names, ₱0 prices, and
  no variety tags. Cards only displayed the business logo fallback image.

ROOT CAUSE:
  API returns `product_name`, `variety_name`, `price`, `is_in_stock` but
  frontend normalizers were reading `name`, `description`, `variety`, `inStock`
  — all undefined from the API response.

SOLUTION:
  Updated normalizers in both pages:

  Home.jsx (lines ~112-120):
    name: p.product_name || p.name
    description: variety_name + " — " + weight_formatted (synthesized)
    tags: [variety_name] if no explicit tags
    price: parseFloat(p.price) || 0

  Products.jsx (lines ~74-86):
    name: product.product_name || product.name
    variety: product.variety_name || product.variety
    description: variety_name + " — " + weight_formatted (synthesized)
    inStock: product.is_in_stock ?? product.in_stock ?? true
