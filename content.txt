  SECTION 1: WHAT EXISTS RIGHT NOW (Current System Scan)
════════════════════════════════════════════════════════════════

── PROCUREMENT ─────────────────────────────────────────────────

  Table: procurements
  Columns: id, supplier_id, variety_id, quantity_kg, sacks,
           price_per_kg, total_cost, description, status,
           created_at, updated_at, deleted_at

  Status values: Pending → Drying → Dried → Completed | Cancelled
  Relationships:
    - belongs to: Supplier, Variety
    - has many:   DryingProcess

  Current behavior:
    - One procurement = one supplier, one variety, one delivery
    - No batch concept. Each row is fully independent.
    - No date grouping. No harvest-season grouping.


── DRYING PROCESS ──────────────────────────────────────────────

  Table: drying_processes
  Columns: id, procurement_id, quantity_kg, sacks, quantity_out,
           days, price (per sack per day), total_price, status,
           created_at, updated_at, deleted_at

  Status values: Drying → Dried | Cancelled
  Relationships:
    - belongs to: Procurement
    - has many:   Processing  (via drying_process_id added by migration)

  Current behavior:
    - One drying record = one procurement only (1-to-1 FK)
    - quantity_out accumulates as processing records consume it
    - Cannot currently merge multiple procurements into one drying batch


── PROCESSING ──────────────────────────────────────────────────

  Table: processings
  Columns: id, procurement_id, drying_process_id, input_kg,
           output_kg, stock_out, husk_kg, yield_percent,
           operator_name, status, processing_date,
           completed_date, created_at, updated_at, deleted_at

  Status values: Pending → Processing → Completed
  Stock status (appended): Pending | Partial | Distributed
  Relationships:
    - belongs to: Procurement, DryingProcess

  Current behavior:
    - A processing can come directly from a procurement (drying skipped)
      OR from a drying process record
    - drying.quantity_out is incremented when processing is created
    - output_kg feeds into product stock (indirectly via stock_out)


── PRODUCT ─────────────────────────────────────────────────────

  Table: products
  Columns: id, category_id (→ varieties.id), name, description,
           price, stock, unit, status, is_featured,
           created_at, updated_at, deleted_at

  Relationships:
    - belongs to: Variety (via category_id column)

  Current behavior:
    - Products are independent (manually managed stock)
    - Processing's stock_out doesn't automatically push to product stock
    - Variety links product to the rice variety type

  SECTION 2: THE REAL-WORLD PROBLEM (Your Explanation)
════════════════════════════════════════════════════════════════

  SCENARIO A — Single Supplier (Clean / 1-to-1 flow)
  ─────────────────────────────────────────────────────
  Supplier gives 500 sacks / 8,000 kg of IR64.
  You process it alone, no mixing with others.

  Flow works perfectly today:
    Procurement #001 (IR64, Supplier A, 500 sacks)
      → Drying #001 (500 sacks)
        → Processing #001 (input 8,000 kg → output X kg)

  ✅ No problem here. It's 1-to-1 all the way.


  SCENARIO B — Multiple Suppliers, Same Variety (The Problem)
  ────────────────────────────────────────────────────────────
  Harvest season: many suppliers come at the same time.
  All bring IR64. All palay goes into the same storage bin/bay.

  Now you can't track which kg came from which supplier.
  When you sell, you just scoop from the IR64 bin.
  You price it as IR64, not by individual supplier.

  ❌ Current system can't represent this.
     You'd have to pick ONE procurement per drying, per processing.
     The "merged bin" reality isn't modeled.


  SCENARIO C — Batch Concept (Your Solution)
  ────────────────────────────────────────────
  You want to group procurements into a BATCH.
  Batch = same variety + same season/date window.

  Example:
    Batch 1 → IR64 (Harvest Season Feb 2026)
      - Procurement #001: Supplier A, 500 sacks
      - Procurement #002: Supplier B, 300 sacks
      - Procurement #003: Supplier C, 200 sacks
      Total: 1,000 sacks of IR64 all in same bin

    Batch 2 → Sinandomeng (Harvest Season Feb 2026)
      - Procurement #004: Supplier D, 400 sacks

  Then you can:
    - Send Batch 1 as a whole to Drying
    - OR still send individual Procurement #001 to Drying separately
    - Process Batch 1 as a whole
    - OR still process individual Drying record separately


════════════════════════════════════════════════════════════════
  SECTION 3: PROPOSED DESIGN
════════════════════════════════════════════════════════════════

  ── CORE CONCEPT ────────────────────────────────────────────

  Add a new "Batch" layer that sits ABOVE Procurement.
  Procurement stays exactly as it is (per-supplier, per-delivery).
  A Batch optionally groups multiple Procurements together.

  This keeps the 1-to-1 flow intact for Scenario A.
  And enables the merged-bin flow for Scenario B.


  ── NEW TABLE: procurement_batches ──────────────────────────

  id
  batch_number       → auto-generated: BATCH-YYYYMMDD-001
                       e.g. BATCH-20260222-001 (date + daily seq)
  variety_id         → FK to varieties (all items in batch = same variety)
  season_date        → date field, defaults to today (no free-text name)
  total_sacks        → denormalized SUM, updated when procurement joins/leaves
  total_kg           → denormalized SUM, updated when procurement joins/leaves
  remaining_sacks    → decremented each time drying draws from this batch
  remaining_kg       → decremented each time drying draws from this batch
  status             → Open | Closed | Completed
  notes              → nullable text
  created_at, updated_at, deleted_at

  ⚡ SPEED NOTE: total_sacks, total_kg, remaining_sacks, remaining_kg are
  stored directly on the row (denormalized). No SUM() query needed at
  display time. Fast O(1) reads for the batch list page.


  ── CHANGE TO: procurements ────────────────────────────────

  Add column:
    batch_id   → nullable FK to procurement_batches
                 NULL = standalone procurement (Scenario A, no batch needed)
                 SET  = this procurement belongs to a batch (Scenario B)

  Rules:
    - A procurement can only belong to ONE batch
    - All procurements in a batch must share the same variety_id
    - ★ CRITICAL: A procurement that belongs to a batch can STILL be
      used individually by its own procurement_id for drying.
      Being in a batch does NOT lock it from individual use.
      The batch is just a grouping/organizational layer.
    - A procurement can be MOVED to another batch while that batch
      is still "Open". Once the batch is "Closed" the assignment is locked.


  ── DRYING: Two entry modes ────────────────────────────────

  Current: drying_processes.procurement_id → one procurement only

  Proposed: keep procurement_id for single-procurement drying
  Add:       batch_id (nullable FK) for batch drying

  BOTH MODES CAN COEXIST:
    - You can dry a single procurement_id even if that procurement
      belongs to a batch. Its own sacks and kg data are used directly.
    - You can dry from a batch (batch_id set), in which case:
        • You choose HOW MANY SACKS to send to drying (partial or full).
        • Drying price is per sack (confirmed).
        • The kg is calculated from those sacks.
        • That amount is SUBTRACTED from the batch's remaining sacks/kg.
        • You do NOT need to dry all sacks at once — space may be limited.
        • Multiple drying records can draw from the same batch over time.

  Rule:  exactly ONE of (procurement_id, batch_id) must be set per record.
         You cannot set both on the same drying record.

  New pivot table: drying_batch_procurements
    id
    drying_process_id  → FK to drying_processes
    procurement_id     → FK to procurements
    sacks_taken        → how many sacks from this procurement in this drying
    quantity_kg        → the kg equivalent of those sacks

  This pivot enables full traceability: for any batch drying record
  you can see exactly which supplier's palay is inside, and how much.


  ── PROCESSING: Two entry modes ────────────────────────────

  Same approach. Processing can come from:
    - A single drying record (current behavior, unchanged)
    - A batch drying record

  No schema change needed here — once drying is resolved,
  processing just picks up from drying as it does today.


  ── SUMMARY OF NEW/CHANGED DB OBJECTS ──────────────────────

  NEW TABLE:
    procurement_batches
      id, batch_number, variety_id, name, season_date,
      total_sacks, total_kg, remaining_sacks, remaining_kg,
      status, notes, created_at, updated_at, deleted_at

      NOTE: remaining_sacks / remaining_kg are decremented each time
      a batch drying record draws from this batch. This lets the UI
      show "how much of this batch is still uncooked".

  MODIFIED TABLE:
    procurements
      + batch_id (nullable FK → procurement_batches, nullOnDelete)

  MODIFIED TABLE:
    drying_processes
      + batch_id (nullable FK → procurement_batches, nullOnDelete)
      (procurement_id and batch_id are mutually exclusive per row)

  NEW PIVOT TABLE (for batch drying traceability):
    drying_batch_procurements
      id, drying_process_id, procurement_id,
      sacks_taken, quantity_kg

  SECTION 4: FLOW DIAGRAMS
════════════════════════════════════════════════════════════════

  FLOW A — Standalone, No Batch (unchanged from today)
  ──────────────────────────────────────────────────────
  Supplier delivers
      ↓
  Procurement #001 (batch_id = NULL)
      ↓
  Drying #001  (procurement_id = #001, batch_id = NULL)
      ↓
  Processing #001  (drying_process_id = #001)
      ↓
  Product/Stock


  FLOW B — Full Batch Drying (new)
  ──────────────────────────────────────────────────────
  Multiple suppliers deliver IR64 same week
      ↓
  Create Batch #001 (IR64, Feb 22 2026)
  [batch_number auto = BATCH-2026-001]
      ↓
  Procurement #002 (batch_id = #001, Supplier A, 500 sacks, 8,000 kg)
  Procurement #003 (batch_id = #001, Supplier B, 300 sacks, 4,800 kg)
  Procurement #004 (batch_id = #001, Supplier C, 200 sacks, 3,200 kg)
      ↓
  Batch totals: 1,000 sacks, 16,000 kg remaining_sacks=1000 remaining_kg=16000
      ↓
  Drying #002  (batch_id=#001, sacks=1000, kg=16000, price per sack)
      [drying_batch_procurements: #002→Proc#002 500sacks, →Proc#003 300sacks, →Proc#004 200sacks]
      [batch remaining_sacks = 0, remaining_kg = 0]
      ↓
  Processing #002  (drying_process_id = #002, input 16,000 kg)
      ↓
  Product/Stock (IR64 rice)


  FLOW C — Partial Batch Drying + KG TRACKING
  ──────────────────────────────────────────────────────
  Batch #001 has 1,000 sacks:
    Proc #002: Supplier A, 500 sacks, 8,000 kg  → 16 kg/sack
    Proc #003: Supplier B, 300 sacks, 4,800 kg  → 16 kg/sack
    Proc #004: Supplier C, 200 sacks, 3,200 kg  → 16 kg/sack
  Dryer only fits 400 sacks today.
      ↓
  Owner enters: "400 sacks from Batch #001"
  System distributes PROPORTIONALLY:
    Proc #002 share: 400 × (500/1000) = 200 sacks → 200×16 = 3,200 kg
    Proc #003 share: 400 × (300/1000) = 120 sacks → 120×16 = 1,920 kg
    Proc #004 share: 400 × (200/1000) =  80 sacks →  80×16 = 1,280 kg
    Total kg this drying = 6,400 kg ✅

  drying_batch_procurements records created:
    { drying_id=3, proc_id=2, sacks_taken=200, quantity_kg=3200 }
    { drying_id=3, proc_id=3, sacks_taken=120, quantity_kg=1920 }
    { drying_id=3, proc_id=4, sacks_taken=80,  quantity_kg=1280 }

  Batch #001 updated: remaining_sacks=600, remaining_kg=9,600
      ↓
  Later: Drying #004 (batch_id=#001, sacks=600, kg=9,600) — same logic
      [batch remaining_sacks = 0, remaining_kg = 0 → auto-Closed]

  ★ DIFFERENT kg/sack ratios work the same way:
    If Proc A=16kg/sack, Proc B=14kg/sack, Proc C=18kg/sack
    → each procurement's own ratio is used in the pivot record kg calc
    → total kg = sum of each proc's proportional contribution


  FLOW D — Procurement in Batch, Used Individually
  ──────────────────────────────────────────────────
  Procurement #003 (batch_id=#001, Supplier B) — different palay quality
  Owner decides to dry it separately, not as part of the batch.
      ↓
  Drying #005  (procurement_id = #003, batch_id = NULL)
      [This uses Procurement #003's own sacks/kg data directly]
      [batch remaining_sacks must account for this — subtract #003's sacks]
      ↓
  Processing #003  (drying_process_id = #005)

  SECTION 6: THINGS TO KEEP IN MIND / CONSTRAINTS
════════════════════════════════════════════════════════════════

  1. BACKWARD COMPATIBILITY
     All existing procurement/drying/processing records have batch_id = NULL.
     They continue to work exactly as before. No data migration needed.

  2. VARIETY CONSTRAINT ON BATCHES
     A batch is tied to exactly one variety.
     Procurement can only join a batch if its variety_id matches.
     This ensures the bin represents one known variety only.

  3. SACKS field in procurement
     The sacks column was added via migration (it exists now).
     Batch totals = SUM(sacks) and SUM(quantity_kg) from member procurements.
     These can be computed on the fly or cached as denormalized columns.

  4. PARTIAL DRYING FROM BATCH
     Flow C shows: a procurement inside a batch can still be dried alone.
     The system must not double-count kg if both single AND batch drying
     exist for the same procurement. Need a "used_in_drying" flag or
     simply track it via drying_batch_procurements pivot records.

  5. STATUS FLOW FOR BATCH
     Open   → all procurements collected, no drying yet
     Closed → drying has started (batch locked, no more adding procurements)
     Processing → drying done, in milling
     Completed  → processing done, rice in stock

  6. PROCUREMENT STATUS WHEN PART OF A BATCH
     Individual procurement status still flows normally:
     Pending → Drying → Dried → Completed
     Even if it's in a batch, each row tracks its own status.

  7. NO FORCED BATCH — IT IS OPTIONAL
     Owner can always do standalone procurement without a batch.
     Batch is a convenience grouping, not a requirement. //yes

  8. AUTO BATCH NUMBER FORMAT
     BATCH-{YYYYMMDD}-{3-digit-seq} e.g. BATCH-20260222-001
     Includes the date of batch creation. Seq resets daily.
     Format is descriptive at a glance (you see when it was created).

  SECTION 8: FINALIZED ANSWERS TO DESIGN QUESTIONS

  Q1. Procurement pricing in a batch?
      ✅ Each procurement keeps its own price_per_kg.
         Batch shows a weighted average for reference only.

  Q2. Batch drying: one record for whole batch OR one per procurement?
      ✅ BOTH are allowed:
         - One drying record for the entire batch (or partial sacks from it).
         - OR individual procurement drying using procurement_id directly,
           even if that procurement belongs to a batch.

  Q3. How to create a batch?
      ✅ Option (a): Inline when adding a procurement.
         - Batch number auto-generated (BATCH-{YEAR}-{SEQ}).
         - User selects/enters the variety name.
         - Season date defaults to today (editable).

  Q4. Batch season field?
      ✅ Use a date field (season_date), defaults to current date.
         Displayed as a formatted date label in the UI.

  Q5. Can a procurement be moved between batches?
      ✅ Yes — movable while batch is "Open".
         Locked once batch is "Closed" (drying has started).
         This allows corrections without data loss.

  Q6. Lock batch when sent to drying?
      ✅ Yes — batch becomes "Closed" when any drying draws from it.
         Reason: dryer space may not fit the whole batch at once,
         so partial drying is allowed (you choose sacks, not all at once).
         Once Closed, no new procurements can be added.

  Q7. Where does "Send to Drying" button appear?
      ✅ BOTH:
         - On Procurement page (Batch view) per batch row.
         - On Drying page (Add Drying modal) as source selector.

  Q8. Batch Profitability summary?
      ✅ Yes — show:
         - Total procurement cost (sum of each procurement's total_cost)
         - Total drying cost (sum of drying records' total_price)
         - Total processing input kg, output kg, yield %
         - Estimated revenue (output kg × target product price)
         - Net profit estimate

════════════════════════════════════════════════════════════════
  SECTION 9: MONEY TRACKING — WHAT EXISTS TODAY
════════════════════════════════════════════════════════════════

  ── STAGE 1: PROCUREMENT (Buying Palay) ───────────────────

  Fields that track money:
    price_per_kg   → decimal(12,2) — unit cost from supplier
    total_cost     → decimal(14,2) — AUTO-CALCULATED on save

  How total_cost is calculated:
    Procurement model boot() → static::saving():
      total_cost = quantity_kg × price_per_kg

  Example:
    Supplier A delivers 8,000 kg at ₱8.50/kg
    total_cost = 8,000 × 8.50 = ₱68,000

  ✅ This works. Money spent on raw palay is tracked per delivery.


  ── STAGE 2: DRYING (Drying Cost) ─────────────────────────

  Fields that track money:
    price          → decimal(12,2) — cost per sack for drying
    total_price    → decimal(14,2) — calculated by recalculateTotal()

  How total_price is calculated:
    DryingProcess::recalculateTotal():
      total_price = (sacks × price) × days

  Example:
    400 sacks dried for 3 days at ₱5/sack/day
    total_price = (400 × 5) × 3 = ₱6,000

  ⚠️ NOTE: recalculateTotal() is NOT auto-called on save.
     It must be called manually after updating sacks/price/days.

  ✅ Drying cost is tracked, but be aware of the manual call.


  ── STAGE 3: PROCESSING (Milling) ─────────────────────────

  Fields that track money:
    ❌ NONE — Zero financial fields

  What exists:
    input_kg, output_kg, husk_kg, yield_percent, stock_out
    operator_name, processing_date, completed_date

  What's missing:
    - Milling fee / milling cost
    - Labor cost
    - Electricity / fuel cost
    - Any cost-per-kg for milling

  ❌ This is a GAP. You cannot calculate total production cost 
     without knowing how much milling costs.

  ✅ OWNER'S DECISION: Milling cost will NOT be tracked separately.
     Instead, the admin directly sets the product selling price.
     The admin looks at total expenses (procurement + drying) per
     batch or standalone, then uses judgment to set a price that
     covers milling, labor, and still makes profit.
     
     This means:
       TOTAL EXPENSES per batch = Procurement total_cost + Drying total_price
       That's it. No processing cost fields needed.
       The admin bakes milling cost into the product price manually.

     Example:
       Procurement: ₱68,000 (8,000 kg × ₱8.50/kg)
       Drying:      ₱6,000  (400 sacks × ₱5 × 3 days)
       ─────────────────────────────────
       Total Expenses: ₱74,000
       Output after milling: 5,600 kg (70% yield)
       Cost per kg of finished rice: ₱74,000 ÷ 5,600 = ₱13.21/kg
       
       Admin sees ₱13.21/kg cost → sets selling price at ₱45/kg
       That includes milling, labor, profit — all in the admin's head.


  ── STAGE 4: PRODUCT (Selling Rice) ───────────────────────

  Fields that track money:
    price          → decimal(10,2) — SELLING price per unit

  What's missing:
    - No link back to which processing record produced it

  ✅ CLARIFICATION on "cannot calculate profit per product":
     This just meant the system doesn't automatically know the 
     production cost. But since you (the admin) manually set the 
     product price AFTER looking at your batch expenses, you already
     factor in your costs mentally.

     What the system CAN do for you:

    - Profit/Loss cannot be calculated yet

  ✅ ANSWER: "How do I calculate profit based on what I have?"
     
     Right now you can only see EXPENSES (money going out):
       Total Expenses = SUM(procurement total_cost) + SUM(drying total_price)
     
     To calculate PROFIT you need REVENUE (money coming in):
       Revenue = quantity_sold × product_price
     
     So you need the Sales module first. Then:
       PROFIT = Total Revenue - Total Expenses
     
     That's it. Simple as that. No processing cost needed since 
     you already bake it into the product price.

  SECTION 10: PRODUCT / PALAY QUANTITY TRACKING — FULL FLOW
════════════════════════════════════════════════════════════════

  ── THE QUANTITY JOURNEY ──────────────────────────────────

  STEP 1: Procurement (Raw Palay Enters System)
  ──────────────────────────────────────────────
    Fields: quantity_kg, sacks
    Status: Pending → Drying → Dried → Completed
    Batch: optional grouping via batch_id → ProcurementBatch

    When sent to drying:
      - Procurement status changes to "Drying"
      - Drying record created with quantity_kg + sacks from this procurement
      - If batch drying: batch's remaining_sacks/remaining_kg decremented

  STEP 2: Drying (Palay Being Dried)
  ──────────────────────────────────────────────
    Fields: quantity_kg, sacks, quantity_out
    Status: Drying → Dried | Cancelled

    quantity_out tracks how much has been sent to processing:
      - Starts at 0
      - INCREMENTED when Processing record is created
      - DECREMENTED when Processing record is deleted
      - remaining_quantity = max(0, quantity_kg - quantity_out)

    Multi-source pivot: drying_batch_procurements
      - Tracks: drying_process_id, procurement_id, sacks_taken, quantity_kg
      - Shows exactly how much from each procurement went into this drying

  STEP 3: Processing (Milling — Palay → Rice)
  ──────────────────────────────────────────────
    Fields: input_kg, output_kg, husk_kg, yield_percent, stock_out
    Status: Pending → Processing → Completed
    Stock Status: Pending | Partial | Distributed (for completed)

    Key calculations (auto on completion):
      husk_kg = input_kg - output_kg
      yield_percent = (output_kg / input_kg) × 100
      remaining_stock = max(0, output_kg - stock_out)

    Multi-source pivot: processing_drying_sources
      - Tracks: processing_id, drying_process_id, quantity_kg
      - One processing can pull from MULTIPLE drying sources
      - Each source's quantity_out is properly incremented/decremented

    stock_out field:
      ⚠️ EXISTS but is NEVER auto-incremented by any service
      ⚠️ Set to 0 on completion via completeProcessing()
      ⚠️ NO automatic distribution to product stock
      ⚠️ This is a PLACEHOLDER for a future feature

  STEP 4: Product (Finished Rice for Sale)
  ──────────────────────────────────────────────
    Fields: stocks (integer), price, unit, weight
    Status: Active | Inactive

    stock management:
      - Manual via ProductController::updateStock endpoint
      - Three modes: add / subtract / set
      - Example: POST /products/{id}/stock { action: 'add', quantity: 100 }

    ❌ NO automatic flow from Processing output_kg → Product stocks
       This is completely manual / disconnected

  SECTION 11: ALL PIVOT / JUNCTION TABLES
════════════════════════════════════════════════════════════════

  ── drying_batch_procurements ─────────────────────────────

  Purpose: When a batch is dried, tracks how much from each
           procurement went into that specific drying record

  Fields:
    id, drying_process_id (FK), procurement_id (FK),
    sacks_taken (int), quantity_kg (decimal), timestamps

  Used by:
    - ProcurementBatch::recalculateTotals() — computes remaining sacks/kg
    - DryingProcessService — creates pivot rows when batch drying is created
    - Proportional distribution: sacks split based on each procurement's
      share of the batch total 


  ── processing_drying_sources ─────────────────────────────

  Purpose: Allows ONE processing record to draw from MULTIPLE
           drying sources (multi-source processing)

  Fields:
    id, processing_id (FK), drying_process_id (FK),
    quantity_kg (decimal), timestamps
    UNIQUE constraint: (processing_id, drying_process_id)

  Used by:
    - ProcessingService::createProcessing() — proportionally splits
      input_kg across selected sources, increments each quantity_out
    - ProcessingService::updateProcessing() — reverses old splits,
      recalculates new proportional distribution
    - ProcessingService::deleteProcessing() — decrements quantity_out
      on each source and detaches pivot rows
