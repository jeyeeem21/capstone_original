2. SOFT DELETE / ARCHIVE STATUS — ALL MODULES
===============================================================================

All delete actions now perform SOFT DELETE. No data is permanently removed 
from the database. Archived records can be restored from Settings → Archives.

┌─────────────────────┬──────────────┬────────────────────────────────────────┐
│ MODULE              │ SOFT DELETE  │ MECHANISM                              │
├─────────────────────┼──────────────┼────────────────────────────────────────┤
│ Products            │ ✅ YES       │ Custom: is_deleted boolean + scopes    │
│ Varieties           │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Suppliers           │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Customers           │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Procurements        │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Procurement Batches │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Drying Processes    │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Processings         │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Drivers             │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Deliveries          │ ✅ YES       │ Laravel SoftDeletes trait              │
│ Users (Staff/Admin) │ ✅ YES       │ Laravel SoftDeletes trait (JUST ADDED) │
│ Sales               │ N/A          │ Uses status lifecycle (voided/returned)│
│ Audit Trail         │ N/A          │ Read-only log, never deleted           │
│ Settings            │ N/A          │ Config records, never deleted          │
│ Website Content     │ N/A          │ CMS content, never deleted             │
└─────────────────────┴──────────────┴────────────────────────────────────────┘

Archive Page: Shows all soft-deleted records from ALL modules above.
              Actions: Restore only. Permanent Delete REMOVED.
              Restored records return to Active status automatically.

6. KNOWN ISSUES / NOTES (Updated March 2026)
===============================================================================

1. DRYING — POSTPONE ACTION
   The backend has a POST /drying-processes/{id}/postpone route but the
   frontend DryingProcess.jsx does not seem to expose a visible "Postpone"
   button in the UI. The backend logic exists and works; it's just not
   triggered from the frontend action buttons currently. Low priority —
   status can be managed via other actions.

2. CATEGORIES.JSX — ORPHANED FILE
   File exists at pages/admin/Products/Categories.jsx but has NO route
   in App.jsx. It duplicates Varieties functionality (uses same /varieties
   API). Can be safely deleted if not needed.

3. MOBILE BOTTOM NAV — MISSING PAGES
   The BottomNav (mobile navigation) includes Dashboard, Procurement,
   Processing, Products, POS, Partners, Staff, Sales, Settings but does
   NOT include Drying and Orders pages. Users must use the sidebar or
   navigate directly on mobile.

4. SALES — NO DELETE ACTION
   Sales use status-based lifecycle (completed → voided / returned) rather
   than soft delete. This is by design — all sales records are always
   preserved as transaction history. Voided sales cannot be "restored" 
   through Archives; voided status is permanent.

5. PRODUCT SOFT DELETE — CUSTOM IMPLEMENTATION
   Product uses is_deleted boolean instead of Laravel's SoftDeletes trait.
   This works correctly with custom global scope and Archive integration,
   but differs from all other models which use the standard trait.

6. DRYING BATCH PROCUREMENTS — PIVOT RECORDS
   When a DryingProcess is archived, its DryingBatchProcurement pivot
   records are hard-deleted before the parent is soft-deleted. If the
   DryingProcess is later restored from Archives, these pivot rows will
   NOT be restored. The drying process record itself is fine, but the
   batch procurement associations are lost.

7. FOOTER QUICK LINKS — ROLE-BASED (Fixed March 2026)
   Footer.jsx now uses useAuth() for role-aware quick links:
   - Staff: Links to /staff/pos, /staff/profile
   - Admin/SuperAdmin: Links to {basePath}/products, /pos, /orders,
     /procurement, /sales
   - Public layout has separate footer with /about, /products, /contact

8. PUBLIC PRODUCT CARDS — FIELD MAPPING (Fixed March 2026)
   Product normalizers in Home.jsx and Products.jsx now correctly map
   API fields: product_name→name, variety_name→variety tags,
   weight_formatted→description, is_in_stock→inStock.
